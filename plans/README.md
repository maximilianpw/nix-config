## Parallel execution strategy (subagent dispatch)

Requested by the operator: split execution across subagents for faster
turnaround and smaller per-agent context. Each plan is written to be
self-contained — a subagent needs only its plan file and the repo.

**Precondition (once, before any dispatch):** commit the current WIP so
worktrees contain fleet.nix/remote-dev.nix (see hazard above).

**Wave 1 — dispatch in parallel, one isolated worktree each:**

| Subagent | Plan | Why parallel-safe |
|----------|------|-------------------|
| A | 003 docs refresh | touches only CLAUDE.md/README/BOOTSTRAP |
| B | 005 fleet + homepage hardening | touches only fleet.nix + homepage.nix |
| C | 006 comma + nix-your-shell | touches only home-manager.nix, dev-tools.nix, shells.nix |
| D | 007 nh pilot | touches only terminal-tools.nix, shells.nix* |

*C and D both add lines to `users/maxpw/modules/shells.nix` — trivially
mergeable (different alias lines), but if you want zero merge work, put D in
Wave 2.

**Wave 2 — the flake.nix chain, one subagent at a time (or one subagent
running all four sequentially):** 002 → 004 → 008 → 009. Rebase/merge each
before starting the next.

**Dispatch prompt template** (per subagent):

> Read `/path/to/repo/plans/NNN-<slug>.md` in full and execute it exactly.
> Run its drift check first. Honor every verification gate and STOP
> condition. Do not modify files outside the plan's in-scope list. Report:
> steps completed, verification outputs, and any STOP condition hit. Do not
> update plans/README.md (the dispatcher maintains the index).

**Review step (dispatcher, after each subagent):** re-run the plan's done
criteria commands yourself, check `git status`/diff for out-of-scope changes,
then update the status table. Steps that require a *switch on real hardware*
(005 step 4, 008 step 3, 009 step 3) cannot be done by a sandboxed subagent —
subagents deliver the eval-verified change; the operator applies and runs the
functional checks.
